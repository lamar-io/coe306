What is an Embedded System?

📌 Definition

A programmable computing system designed to perform a dedicated function as part of a larger product. Takes advantage of application characteristics to optimize design — doesn't need general-purpose features.

Examples: calculator, microwave, printer, car (40–100 MCUs), GPS moving map display.

🔄 The Modern Spectrum

Classification is no longer binary — every smart device sits somewhere on the spectrum:

  • Old embedded: simple MCU, no OS, assembly
  • Modern embedded: can run Linux, touch UI
  • General purpose: multi-core, full OS, user-programmable

Portables got more powerful; embedded devices gained user interfaces and networking.

Characteristics of Embedded Systems

⏱️ Real-Time Requirements

Performance = worst-case, not average. Must finish operations by deadlines.

  • Hard: deadline miss = system failure (ABS brakes, printer page boundary, medical pump)
  • Soft: miss = degraded performance (video streaming drops frames)
  • Multi-rate: multiple operations at widely varying rates simultaneously

💡 Key Characteristics

  • Sophisticated functionality: complex algorithms, moving-map GPS
  • Low manufacturing cost: mass-market pressure
  • Low power: critical for battery devices; excess power → heat
  • Tight deadlines: small teams, 6-month market windows
  • Upgradeability: OTA firmware updates

📊 Performance Analysis Levels

Unlike general purpose (avg throughput), embedded = deadline meeting. Must analyze at:

  • Instruction: CPU microarchitecture effects
  • Platform: I/O and memory subsystem
  • Program: overall structure and behavior
  • Task: RTOS scheduling and interaction
  • Multiprocessor: processor interaction

🖥️ Embedded Software Considerations

  • Peripherals: timing, response, data types, power, bandwidth
  • Communication: bandwidth, reliability, redundancy
  • Real-time performance and scheduling
  • Available compute, memory, toolchains
  • Security and privacy
  • Sensor/actuator interfacing
  • Testing under real-world noise and jitter

Design Process

📋 Embedded System Design Flow

1
Requirements — plain-language description of what the user expects, independent of implementation.
• Functional: what the system does (e.g., "display current position on map")
• Non-functional: how well (e.g., "GPS lock within 15 seconds", "power-up in 1 second")
• Constraints: restrict design choices (e.g., "use 400×600 display", "fits in hand", "$120 price")
2
Specification — formal, precise, consistent, unambiguous description. Must be: complete and verifiable. Bridges "what user wants" ↔ "what gets built."
3
Architecture Design — describes how the system implements the spec. Defines H/W + S/W component structure (block diagrams: general, hardware, software architectures).
4
Component Design — build individual modules. Some ready-made (CPU, GPS receiver, map databases), some custom (PCB, FPGA, software modules).
5
System Integration — most bugs appear here. Component interactions reveal issues invisible in isolation.
• Verification: "Did we build the system right?" (matches spec?)
• Validation: "Did we build the right system?" (does what user actually needs?)

⚠️ Design Challenges

  • Deadlines must be met in worst case
  • Power/energy: sleep modes, clock gating, DVFS
  • Resource constraints: performance vs cost
  • Limited debugging: JTAG, UART logs, oscilloscopes
  • Real-world noise, temperature variation
  • Testing: jitter, race conditions, timing bugs
  • Correctness under uncertainty

💾 SPSR — Saved Program Status Register

Each privileged mode (except System) has its own SPSR.

  • On exception entry: SPSR ← CPSR (saves current state)
  • On return: CPSR ← SPSR (restores user state)
  • System mode has no SPSR — uses User register set
  • If a nested exception of the same mode occurs, SPSR must be saved manually to support nested exceptions.

Barrel Shifter & Immediate Values

🔄 Barrel Shifter — Part of Every Data Instruction

ARM has no dedicated shift instructions. Instead the second operand passes through a barrel shifter before the ALU. Shifts are free inside any data processing instruction.

ShiftMeaningNotes
LSL #nLogical shift left n (×2ⁿ)Zero-fills right
LSR #nLogical shift right n (÷2ⁿ unsigned)Zero-fills left
ASR #nArithmetic shift right (sign-preserving)Sign-fills left
ROR #nRotate right n bitsBits wrap around
RRXRotate right 1 bit through carry33-bit rotation

Shift aliases are MOV shorthands: LSL r0,r1,#3 ≡ MOV r0,r1,LSL #3

ADD r0, r1, r2, LSL #2    ; r0 = r1 + r2×4
SUB r3, r4, r5, ASR #1    ; r3 = r4 - r5/2
MOV r0, r0, RRX            ; rotate right thru C

🔢 Multiplying by Constants — Barrel Shifter Tricks

Multiplications by constants = (power of 2) ± 1 can be done in one cycle. Combine with RSB for factors of 7, 15, 31, etc.

; r0 = r1 × 5  →  5 = 4+1
ADD r0, r1, r1, LSL #2

; r0 = r0 × 7  →  7 = 8-1
RSB r0, r0, r0, LSL #3

; r0 = r0 × 35 = 5×7
ADD r0, r0, r0, LSL #2  ; ×5
RSB r0, r0, r0, LSL #3  ; ×7

; r2 = r3 × 105 = 15×7
RSB r2, r3, r3, LSL #4  ; r2 = r3×15
RSB r2, r2, r2, LSL #3  ; r2 = r2×7

🔢 Immediate Value Encoding — "8-bit rotated right by even number"

ARM instructions are fixed 32-bit. Only 12 bits for Operand2: 8-bit value + 4-bit rotation (×2 → 0–30). Not all 32-bit constants can be encoded; those requiring more use a literal pool.

; Valid constants (8-bit ror even): 0–255 [0x00–0xFF] 256,260,…,1020 [0x40–0xFF ror 30] 1024,1040,…,4080 [0x40–0xFF ror 28] ; Load using rotation explicitly: MOV r0, #0x40, 26 ; = 0x1000 = 4096 MOV r0, #4096 ; assembler converts MVN r0, #0 ; = 0xFFFFFFFF
; Large constants → literal pool: LDR r0, =0xFF ; → MOV r0, #0xFF LDR r0, =0x55555555 ; → LDR r0, [PC,#imm] ; DCD 0x55555555 ; Load address of variable: ADR r1, FOO ; PC-relative, ±4KB LDR r1, =FOO ; always works

Branches, Control Flow & Conditional Execution

🔀 Branch Instructions

B{cond} labelBranch. PC-relative ±32 MB range.
BL{cond} labelBranch with Link: LR ← PC+4, then jump. Used for function calls.
MOV pc, lrReturn from function.
BX RmBranch and exchange (can switch ARM↔Thumb).

✅ Conditional Execution — Every ARM Instruction

ARM's key feature: predication via 4-bit condition field in every instruction. Avoids pipeline-stalling branches for short if-statements. Preserves pipeline at cost of 1 cycle per skipped instruction (vs branch penalty).

; if (r0 != 5) { r1=r1+r0; r1=r1-r2; }
CMP  r0, #5
ADDNE r1, r1, r0
SUBNE r1, r1, r2

; ADDS sets flags too:
ADDS r0, r1, r2    ; r0=r1+r2 AND set N,Z,C,V

📋 Conditional Execution Patterns

; if (a==0) func(1)  [a in r0]
CMP  r0, #0
MOVEQ r0, #1
BLEQ  func

; if (a==0) x=0; if (a>0) x=1
CMP   r0, #0
MOVEQ r1, #0
MOVGT r1, #1

; OR: if (a==4 || a==10) x=0
CMP   r0, #4
CMPNE r0, #10
MOVEQ r1, #0

; AND: if (a==4 && b==10 && c==5) x=0
CMP   r0, #4
CMPEQ r1, #10
CMPEQ r2, #5
MOVEQ r3, #0

; GCD (Euclidean, no branches)
loop: CMP  r0, r1
      SUBGT r0, r0, r1
      SUBLT r1, r1, r0
      BNE   loop

🔄 Control Flow: If/Else with Branches

; C: if (a>b) { x=5; y=c+d; } else x=c-d;
LDR r0, [r4]    ; a
LDR r1, [r4]    ; b
CMP r0, r1
BLE fblock      ; if a<=b → false block
; true block
MOV r0, #5  ; STR x
LDR r0,[r4] ; c
LDR r1,[r4] ; d
ADD r0,r0,r1 ; STR y
B after
fblock:
LDR r0,[r4] ; c
LDR r1,[r4] ; d
SUB r0,r0,r1 ; STR x
after: ...

; Switch statement via jump table
LDR r0, [r2]            ; test value
ADR r1, switchtab
LDR r1, [r1, r0, LSL#2] ; index table
MOV pc, r1              ; jump
switchtab: .word case0, case1 ...

🔁 For Loop (FIR Filter Pattern)

; C: for(i=0,f=0; i<N; i++) f+=c[i]*x[i]
MOV r0, #0       ; i=0
MOV r8, #0       ; array byte index
LDR r1, [r2]     ; N
MOV r2, #0       ; f=0
ADR r3, c
ADR r5, x
loop:
  LDR r4, [r3,r8] ; c[i]
  LDR r6, [r5,r8] ; x[i]
  MUL r4, r6, r4
  ADD r2, r2, r4
  ADD r8, r8, #4  ; advance ptr
  ADD r0, r0, #1  ; i++
  CMP r0, r1
  BLT loop

PIC16F Addressing Modes — Worked Examples

🔹 Direct Addressing

7-bit register address in the instruction selects register in current bank. Bank selected by RP1:RP0 in STATUS.

MOVWF 0x25    ; write W to reg 0x25, current bank
ADDWF 0x25,W  ; W = W + reg[0x25]
INCF  0x30,F  ; reg[0x30]++, store back to F
DECF  0x30,W  ; W = reg[0x30]-1, reg[0x30] unchanged

🔹 Indirect Addressing via FSR+IRP

FSR (8 bits) + IRP bit = 9 bits → 512 addressable locations. Writing to INDF accesses mem[FSR] automatically.

MOVLW 0x55     ; W = 0x55
MOVWF FSR      ; FSR = 0x55 (pointer)
MOVLW 0xAA
MOVWF INDF     ; mem[0x55] = 0xAA
; Read back:
MOVF  INDF,W   ; W = mem[FSR] = mem[0x55]

🔹 PC Absolute (GOTO / CALL)

GOTO/CALL encode 11 bits. Upper 2 bits of 13-bit PC come from PCLATH[4:3]. Must set PCLATH to cross 2K page boundaries.

GOTO 0x200     ; PC[10:0]=0x200; PC[12:11] from PCLATH[4:3]
CALL my_func   ; push PC+1, jump

🔹 Computed GOTO — Jump Table

When ADDWF PCL,F executes at address N, PC already holds N+1. Result = N+1+W.

    org  0x010
    MOVLW 0x03       ; addr 0x010
    ADDWF PCL,F      ; addr 0x011 → PCL=0x012+3=0x015
    NOP              ; 0x012
    NOP              ; 0x013
    NOP              ; 0x014
    MOVLW 0x99       ; 0x015 ← jumps here

🔹 RRF — 9-bit Rotate Right Through Carry

Carries the entire 9-bit value {C, reg[7:0]} right by 1. Old bit[0] → C; old C → new bit[7].

Before: C=0, reg=1011 0100 (0xB4)
After RRF:
  new bit7 = old C = 0
  reg → 0101 1010 (0x5A)
  new C = old bit0 = 0

; Extract 2 bits at a time from byte:
RRF byte,W  ; get bits 1:0 via carry + bit7
RRF byte,W  ; get bits 3:2

🔹 DECFSZ Loop

Standard PIC count-down loop. DECFSZ decrements and skips the next instruction if result = 0.

    MOVLW 0x05
    MOVWF COUNT      ; COUNT = 5
LOOP:
    ...              ; loop body
    DECFSZ COUNT,F   ; COUNT--; skip if 0
    GOTO LOOP        ; repeat
    ...              ; after loop

GOTO executes N−1 times for initial value N. (Skipped on the Nth decrement to zero.)

I/O Devices & Polling

🔌 I/O Device Interface

Typical I/O devices expose three register types:

  • Data registers: hold input/output data exchanged with device
  • Status registers: indicate device state (ready, busy, error, flags)
  • Control registers: configure and command the device

Example: 8251 UART — configures baud rate, parity, stop bits, character width.

🗺️ Memory-Mapped vs Port-Mapped I/O

Memory-mappedDevice registers at memory addresses. CPU uses normal LDR/STR. Common in ARM and most RISC architectures.
Port I/O (isolated)Separate address space. Special IN/OUT instructions. Used in x86.

⏳ Busy-Wait I/O (Polling)

Repeatedly read status register until device ready. Simple but wasteful.

// Write a char:
*OUT_CHAR = c;
*OUT_STATUS = 1;           // start
while(*OUT_STATUS != 0);   // wait

// Copy input→output:
while(1) {
  while(*IN_STATUS == 0);  // wait
  c = *IN_DATA;
  *IN_STATUS = 0;          // ack
  *OUT_DATA = c;
  *OUT_STATUS = 1;
  while(*OUT_STATUS != 0);
}

Interrupt Systems

📋 Complete Interrupt Event Sequence

1
Interrupt source asserts request — device raises interrupt line
2
CPU finishes current instruction — never interrupts mid-instruction; checks at instruction boundaries
3
CPU acknowledges interrupt — signals to device/controller
4
Hardware supplies interrupt vector — interrupt controller or device sends vector number; CPU indexes the interrupt vector table
5
CPU saves execution state — PC, flags, mode-specific registers
6
CPU jumps to ISR — PC = handler address from vector table
7
ISR executes — services the device, clears the interrupt flag
8
Return-from-interrupt restores state
9
Foreground program resumes

⚡ Interrupt Priorities, Masking & Vectors

MaskingLower-priority interrupt not serviced while higher is being handled. Hardware tracks current priority level.
NMINon-Maskable Interrupt — highest priority, cannot be disabled. For critical hardware events (power failure, hardware fault).
Non-vectored / sharedMultiple devices share one interrupt line. ISR must poll each device's status to identify source. Earlier checks = higher software priority.
VectoredDevice sends vector number → CPU indexes table directly to correct ISR. Faster dispatch.
Additional levelsExternal hardware: interrupt controllers, priority encoders, daisy chains.
Vectored entry address: addr = base + (vector_num × entry_size) Example: base=0x80001000, entry_size=4, vector=5 addr = 0x80001000 + 20 = 0x80001014

⏱️ Interrupt Timing Calculation

When a higher-priority interrupt preempts a running handler, use this formula:

finish_cycle = resume_cycle + remaining_cycles − 1
1
Find cycles already run: ran = interrupt_cycle − start
2
Remaining = total_cycles − ran
3
Higher-priority handler runs to completion
4
Preempted resumes next cycle after. Apply formula.
Quiz 3-181: A(30cy, higher) preempts B(25cy) at end of cycle 5
B ran: 5 cycles → remaining = 20
A runs: cycles 6–35 (30 cycles)
B resumes: cycle 36
B finishes: 36 + 20 − 1 = cycle 55

🤖 ARM7 IRQ Response (Responding)

①
LR ← PC + 4 — save return address (PC was 2 ahead due to pipeline; +4 to point to instruction after the interrupted one)
②
SPSR ← CPSR — save current processor state
③
CPSR mode bits ← IRQ — switch to IRQ mode
④
CPSR[I] ← 1 — disable further IRQs
⑤
PC ← 0x00000018 — jump to IRQ vector entry

Leaving ISR: SUBS PC, LR, #4 → restores CPSR from SPSR_irq and returns.

Cache Memory

💾 Cache Basics

  • Small, fast memory storing copies of recently accessed main memory blocks (cache lines)
  • May be: instruction cache + data cache (split/Harvard), or unified
  • Cache hit: block found → fast
  • Cache miss: not found → fetch from main memory → slow
  • Working set: set of locations repeatedly accessed; cache most effective when it fits
  • Caches increase average speed but increase variability

❌ Three Types of Cache Misses

Compulsory (cold)First-ever access to a block — unavoidable. Occurs regardless of cache size.
CapacityWorking set exceeds cache size — too many blocks needed.
ConflictMultiple blocks compete for same cache line in direct-mapped or set-associative caches.

📐 AMAT Formulas

Single-level:

t_avg = t_cache + (1−h) × t_main

Two-level L1+L2:

t_avg = tL1 + (1−h1)tL2 + (1−h1)(1−h2)tmain

Always pay tL1. Only pay tL2 if L1 misses. Only pay tmain if both miss.

🔢 Cache Organizations & Policies

Direct-mappedEach block maps to exactly one cache line → fastest, highest conflict misses
Fully-associativeAny block can go anywhere → fewest misses, most complex
N-way set-assocEach block maps to a set, can go in any of N ways → balance
Write-throughUpdate main memory on every write → simpler but more traffic
Write-backUpdate main memory only when dirty line evicted → less traffic
ReplacementLRU (best), Random, FIFO — which line to evict on miss

3.4

Memory Management

📦 Virtual Memory Overview

Virtual memory provides each process its own virtual address space, independent of physical memory layout. The MMU translates virtual → physical addresses and enforces memory protection.

Advantages:

  • Flexibility: memory can be relocated/resized transparently
  • Abstraction: each process sees contiguous private address space
  • Efficiency: physical memory used efficiently via demand paging
  • Protection: isolate processes; protect kernel from user programs
  • Sharing: shared libraries, shared memory, copy-on-write

MMU tasks:

  • Virtual-to-physical address translation
  • Memory protection and access control
  • Detection of page faults
  • Support for demand paging (pages in swap space)

Page fault: CPU accesses a virtual page not resident in physical memory → OS handler loads it from disk.

📦 Segmentation

Memory divided into variable-size segments. Each segment: base address + limit (size). Provides protection with variable-sized regions, but suffers from fragmentation.

Physical address = segment_base + offset Constraint: offset < segment_limit (else → protection fault)

Segment number N starts at address N × 0x10 (segments at 16-byte boundaries).

Quiz 3-252: Segment #8 allocated 8 KB. Next available segment?
Base of #8: 8 × 0x10 = 0x80
Size 8 KB: 0x2000
End addr: 0x80 + 0x2000 = 0x2080
Next seg #: 0x2080 ÷ 0x10 = 0x208

📄 Paging — Step-by-Step Method

Fixed-size pages in virtual space map to fixed-size frames in physical memory. The OS maintains a page table per process; the MMU does translation using it. Offset bits are identical in virtual and physical addresses.

Virtual = [ Page Number | Offset ] Physical = [ Frame Number | Offset ] (offset unchanged)
1
offset_bits = log₂(page_size_bytes)
4 KB → log₂(4096) = 12 bits
2
page_number_bits = total_address_bits − offset_bits
32-bit addr, 12 offset → 20 bits
3
flat table entries = 2^(page_number_bits)
20 bits → 1,048,576 = 1M entries
4
frame_number_bits = log₂(physical_memory) − offset_bits
1 GB physical → log₂(1GB) = 30, so 30 − 12 = 18 bits per entry
5
Decompose virtual address:
page# = vaddr >> offset_bits
offset = vaddr & (page_size − 1)
Addr 0x00ABC123: page# = 0x00ABC, offset = 0x123
6
Physical address: concatenate [frame#][offset]
Frame 0x02000, offset 0x123 → phys = 0x02000123
Full Example — 32-bit, 4 KB pages, 1 GB physical
offset bits: 12 | page# bits: 20 | entries: 1M | frame bits: 18
addr 0x00ABC123: page#=0x00ABC, offset=0x123
physical (frame 0x02000): 0x02000123

🗂️ Two-Level Page Tables

A flat page table for 32-bit addresses with 4 KB pages = 4 MB per process — wasteful for sparse address spaces. Multi-level tables only allocate second-level tables for used address ranges.

Virtual = [ L1_index | L2_index | Offset ] L1 table: always 1, always allocated L2 tables: 1 per used L1 entry (allocated on demand)

Pentium (x86) 2-level example: 10-bit L1, 10-bit L2, 12-bit offset. L1 = 1024 entries, each points to L2 table of 1024 entries.

1
L2 entries per table = total_flat_entries ÷ L1_entries
2
Pages needed by program = program_size ÷ page_size
3
L2 tables needed = ⌈pages_needed ÷ L2_entries_per_table⌉
4
Total tables = 1 (L1) + L2_tables_needed
Quiz 3-252: 4KB pages, 1024 L1 entries, 20 MB program
L2 entries each: 1M ÷ 1024 = 1024 (covers 4MB)
Pages for 20MB: 20MB ÷ 4KB = 5120
L2 tables: ⌈5120 ÷ 1024⌉ = 5
Total: 1 + 5 = 6 page tables

⚡ TLB — Translation Lookaside Buffer

Page tables are in main memory → every address translation needs a memory access. The TLB is a hardware cache of recent page table entries (virtual → physical mappings) that dramatically speeds up address translation.

TLB hitTranslation found in TLB → very fast (same cycle)
TLB missWalk the page table in memory → slow. Update TLB with the result.
PTE contentsFrame number + Present bit + Dirty bit + Permission bits (R/W/X)

CPU Performance & Power

⚡ CPU Performance

Execution Time = IC × CPI × T_cycle
ICInstruction count
CPICycles per instruction (ARM7 pipeline: 3)
T_cycleClock period = 1/frequency
LatencyCycles for one instruction to complete
Throughput (IPC)Instructions completed per cycle
Pipelining↑ throughput, latency unchanged. Pipeline stalls (bubbles) from: data hazards, control hazards, memory latency

🔋 CMOS Power Consumption

P_total = P_dynamic + P_leakage P_dynamic = α × C × V² × f
α (alpha)Switching activity — fraction of transitions
CLoad capacitance
V²Supply voltage — quadratic! Halving V → ¼ power
fClock frequency — linear
LeakageTransistor leakage; increases with process scaling; eliminated by power-gating

💤 Power-Saving Strategies

  • Lower V (DVFS): quadratic reduction in dynamic power
  • Lower f: linear reduction
  • Clock gating: reduce α by not clocking idle units
  • DVFS: dynamically adjust both V and f per workload
  • Power gating: disconnect idle blocks (eliminates leakage)
  • Sleep modes: halt most chip activity
  • Static PM: fixed user-selected mode (e.g., low-power mode)
  • Dynamic PM: OS adjusts automatically based on workload

💡 PowerPC 603 Power Example

Consumes 2.2W at 80 MHz. Techniques used:

  • Clock gating of unused execution units
  • Unused pipeline stage clock gating
  • 8 KB cache divided into 8 subarrays — at most 2 subarrays accessed per cycle (reduces switching capacitance)

SPEC benchmarks showed high idle rates → dynamic clock gating highly beneficial.
